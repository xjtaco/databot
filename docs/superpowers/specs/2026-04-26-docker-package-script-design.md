# Docker Package & Install Script Design

## Summary

Replace `docker/export.sh` with `docker/package.sh`. The new script builds Docker images via `docker compose build`, then packages images, configuration, and an `install.sh` into a single `tar.gz` archive. The bundled `install.sh` supports `--install-dir` for installation to a specified directory, with automatic upgrade detection and service restart.

## package.sh

### Usage

```bash
./package.sh [-h] [--with-data] [output_dir]
```

### Flow

1. `docker compose build` — build all service images from `docker-compose.yaml`
2. `docker save` — save built images to `images.tar`
3. Collect configuration files (`docker-compose.yaml`, `nginx.conf`, `.env`, `backend/.env`)
4. Generate `install.sh` and embed it in the package
5. Optionally collect `.data/databot/` data directories (`--with-data`)
6. Create `databot-package-YYYYMMDD_HHMMSS.tar.gz`

### Key Changes from export.sh

- **Added `docker compose build`** before image export
- **Dynamic image list** — parse from `docker-compose.yaml` instead of hardcoding
- **Generated script renamed** from `load.sh` to `install.sh`
- **Output filename** changed from `databot-export-*` to `databot-package-*`
- **Removed** the old `export.sh` file

### Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `--with-data` | off | Include `.data/databot/` in the package |
| `output_dir` | `.` | Where to write the `tar.gz` file |

## install.sh (generated by package.sh)

### Usage

```bash
./install.sh --install-dir /opt/databot [databot-package-*.tar.gz]
```

- `--install-dir` is required (no default — must be explicit)
- If `tar.gz` path is provided, extract from the archive
- If no `tar.gz` path, assume running from an already-extracted package directory

### Install Directory Layout

```
<install-dir>/
├── docker-compose.yaml
├── nginx.conf
├── .env                  # docker environment variables
├── backend.env           # backend environment variables
├── images.tar            # Docker images
├── install.sh            # the install script itself
└── data/                 # only present when packaged with --with-data
    ├── logs/
    ├── workfolder/
    ├── dictionary/
    ├── knowledge/
    ├── uploads/
    └── pg_data/
```

All files are placed directly in `install-dir` — no nested `docker/` subdirectory.

### Fresh Install Flow

1. `docker load -i images.tar`
2. Write configuration files (`docker-compose.yaml`, `nginx.conf`, `.env`, `backend.env`)
3. If `data/` exists in the package and no `.data/` at target, restore data
4. `docker compose up -d`
5. Print summary with reminders to configure API keys, JWT secrets, etc.

### Upgrade Flow

Upgrade is detected when `docker-compose.yaml` exists in `install-dir` and `docker compose ps` shows running databot containers.

1. `docker compose down` — stop existing services
2. `docker load -i images.tar` — load new images
3. **Force replace** all configuration files (`docker-compose.yaml`, `nginx.conf`, `.env`, `backend.env`)
4. **Skip data directory** — do not overwrite existing `.data/`
5. `docker compose up -d` — start services with new images and config
6. Print upgrade summary

### Upgrade Detection

A previous installation is detected when BOTH conditions are true:
- `docker-compose.yaml` exists in `install-dir`
- `docker compose ps` (within `install-dir`) returns running containers

## Testing

Replace `docker/tests/export_load_test.sh` with `docker/tests/package_install_test.sh` using the same stub-based approach.

### Test Cases

1. **Help output** — `package.sh -h` prints usage with `--with-data` and `--install-dir` descriptions
2. **Default package without data** — runs build + save + config collection, produces `tar.gz` without `data/`
3. **Package with data** — `--with-data` includes `.data/databot/` subdirectories in `tar.gz`
4. **Fresh install** — extracts to `install-dir`, loads images, writes config, starts services
5. **Upgrade install** — detects existing running services, stops them, replaces images/config, restarts
6. **Data skip on upgrade** — existing `.data/` is not overwritten during upgrade
7. **Install from extracted directory** — `install.sh --install-dir /path` works without a tar.gz argument

### Stub Strategy

Extend existing stubs:
- Add `docker compose` subcommand stubs (`build`, `up`, `down`, `ps`)
- Track called commands via temp files for assertion
- Keep `docker image inspect`, `docker save`, `docker load` stubs
